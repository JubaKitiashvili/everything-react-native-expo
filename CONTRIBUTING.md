# Contributing to ERNE

ERNE (Everything React Native & Expo) is an AI coding agent harness for React Native and Expo development. This guide covers how to contribute agents, commands, hooks, and code.

## Getting Started

```bash
git clone https://github.com/JubaKitiashvili/everything-react-native-expo.git
cd everything-react-native-expo
npm install
npm test
npm run validate
```

Requires Node.js >= 18. No native dependencies needed for the harness itself.

## Project Structure

```
agents/           # Specialized AI agent definitions (.md with frontmatter)
  variants/       # Stack-specific agent variants (generated by init)
commands/         # Slash command definitions (/plan, /tdd, /code-review, etc.)
hooks/            # Hook configuration and profiles
  hooks.json      # Master hook registry
  profiles/       # minimal.json, standard.json, strict.json
scripts/          # Build scripts, validators, hook runners
  hooks/          # Hook implementation scripts
  hooks/lib/      # hook-utils.js shared utilities
lib/              # Core library code (detect, generate, CLI logic)
dashboard/        # Real-time agent dashboard (WebSocket-based)
docs/             # Documentation (getting-started, agents, commands, hooks)
skills/           # Reusable knowledge skills (coding-standards, tdd-workflow, etc.)
contexts/         # Behavior modes (dev, review, vibe)
rules/            # Coding standard rules (layered by platform)
schemas/          # JSON schemas for validation
examples/         # Example configurations
tests/            # Test files (node:test runner)
  hooks/          # Hook-specific tests
bin/              # CLI entry point (erne)
```

## Creating Agents

Each agent is a Markdown file in `agents/` with YAML frontmatter and structured sections.

### Frontmatter (required)

```yaml
---
name: agent-name
emoji: "icon"
vibe: "One-line personality tagline"
description: What the agent does and when it activates.
---
```

- `name` — kebab-case identifier
- `emoji` — single emoji for dashboard/logs
- `vibe` — personality summary (shown in agent listings)
- `description` — capabilities and trigger context (which commands invoke it)

### Required Sections

1. **Identity & Personality** — The agent's perspective and behavioral traits. Make it distinct. An architect thinks in systems; a code reviewer thinks in evidence. Personality drives how the agent communicates, not just what it knows.

2. **Communication Style** — Concrete rules for how the agent talks. Lead with constraints, not solutions. Name trade-offs explicitly. Use bullet points with examples.

3. **Success Metrics** — Quantifiable outcomes the agent optimizes for (e.g., "prop drilling depth: <3 levels", "zero false positives in security findings").

4. **Memory Integration** — Three subsections:
   - *What to Save* — decisions, patterns, and findings worth remembering
   - *What to Search* — queries to run before starting work
   - *Tag Format* — `[agent-name, {project}, category]`

### Additional Sections

Most agents also include: Your Role, Capabilities, Process, Guidelines, and Output Format. See `agents/architect.md` for a complete reference.

### Scaffolding Command

```bash
erne add-agent
```

This runs an interactive prompt to generate a new agent file with the correct structure.

### Design Patterns

From the agency-agents migration, these patterns make agents effective:

- **Personality over instructions** — An agent that "sees systems as living organisms" behaves differently than one told to "analyze architecture." Personality creates consistent, non-generic output.
- **Evidence-based review** — Agents should cite specific files, line numbers, and metrics. No vague advice.
- **Memory integration** — Every agent should save decisions and search past context before starting work. This prevents repeated mistakes and maintains project consistency.

## Creating Commands

Commands are Markdown files in `commands/` with frontmatter:

```yaml
---
name: command-name
description: What this command does
---
```

The body describes the process the AI should follow when the command is invoked. Commands typically delegate to one or more agents. See `commands/plan.md` for reference.

Register new commands in the "Available Commands" section of `CLAUDE.md`.

## Creating Hook Scripts

Hooks are quality-enforcement scripts that run at specific lifecycle events (SessionStart, PostToolUse, etc.). They live in `scripts/hooks/`.

### hook-utils.js API

Import from `scripts/hooks/lib/hook-utils.js`:

| Function | Description |
|---|---|
| `readStdin()` | Parse JSON from stdin (hook input payload) |
| `getEditedFilePath(input)` | Extract file path from tool input |
| `pass(msg)` | Exit 0 — hook passes |
| `fail(msg)` | Exit 1 — hook fails (blocks action) |
| `warn(msg)` | Exit 2 — hook warns (non-blocking) |
| `isTestFile(filePath)` | Check if path is a test/spec file |
| `hasExtension(filePath, exts)` | Check file extension against a list |

### Adding a Hook

1. Create the script in `scripts/hooks/` (e.g., `check-something.js`)
2. Register it in `hooks/hooks.json`:
   ```json
   {
     "event": "PostToolUse",
     "pattern": "Edit|Write",
     "script": "check-something.js",
     "command": "node scripts/hooks/run-with-flags.js check-something.js",
     "profiles": ["standard", "strict"]
   }
   ```
3. Choose which profiles include it: `minimal`, `standard`, `strict`
4. Add a test in `tests/hooks/`

### Hook Events

- `SessionStart` — runs once when a session begins
- `PostToolUse` — runs after a tool call (filter with `pattern`)

## Code Style

### Language and Components

- TypeScript strict mode for all application code
- Functional components only with `const` + arrow functions
- Named exports only (no default exports)
- PascalCase for components, camelCase for hooks/utils, SCREAMING_SNAKE_CASE for constants
- Group imports: react, react-native, expo, external libs, internal modules, types

### Commit Messages

Use [Conventional Commits](https://www.conventionalcommits.org/):

```
feat: add performance-profiler agent
fix: correct deep link parameter validation in hook
refactor: extract hook-utils into shared module
test: add coverage for session-start hook
docs: update agent creation guide
chore: bump node engine requirement
```

One logical change per commit. Keep commits atomic.

## Testing

ERNE uses the built-in `node:test` runner. No external test framework needed.

### Running Tests

```bash
npm test                    # all tests
node --test tests/cli.test.js  # single file
```

### Where Tests Live

- `tests/*.test.js` — core tests (CLI, detection, generation, etc.)
- `tests/hooks/*.test.js` — hook-specific tests

### What to Test

- Every new hook script needs a corresponding test
- Test the behavior, not the implementation
- For agents and commands (Markdown files), validation runs via `npm run validate`

## Pull Request Process

### Branch Naming

```
feat/add-security-agent
fix/hook-stdin-parsing
refactor/extract-detection-logic
```

### Before Submitting

1. Run `npm test` and `npm run validate` — both must pass
2. If adding an agent: verify frontmatter has all required fields (name, emoji, vibe, description)
3. If adding a hook: include the `hooks.json` registration and a test file
4. If adding a command: update the Available Commands list in `CLAUDE.md`

### What Reviewers Check

- **Correctness** — Does it work? Are edge cases handled?
- **Consistency** — Does the agent/command/hook follow established patterns?
- **Atomic commits** — One logical change per commit, conventional commit format
- **Test coverage** — New functionality has tests
- **No hardcoded secrets** — Environment variables for all sensitive values
- **Agent quality** — Personality is distinct, metrics are quantifiable, memory integration is present
